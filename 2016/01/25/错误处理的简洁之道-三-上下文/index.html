错误处理的简洁之道 (三) 上下文

发表于 2016-01-25 更新于 2021-08-16 分类于 tech

debug一般有两种形式，使用调试工具断点单步, 或者分析trace信息。
这里说的trace包括调用栈，调用参数等运行时的上下文信息，它们可以在不影响功能地前提下从实际环境中无痛收集。这两种形式就好比一个是开膛破肚的外科手术，一个是望闻问切的推理分析。前者固然简单直观，但一般仅限于开发环境，如果问题只出现于实际环境，那就只能借助trace来找出真凶。
因此，错误处理的要求之一就是提供尽可能多的上下文信息，以便快速定位问题。

trace object

传统的trace输出是通过函数调用方式，比如上面演示的std::cout和更普遍的类似printf(const char *, …)的格式化输出方式。
相比将trace内容通过构造一个object来输出的方式，后者的优点就是可以控制输出的时机，我们可以在object构造的时候就输出trace，也可以在析构的时候，或者仅在异常回滚时的析构中。
这不就是scope guard么 …

当异常引起的stack unwinding到某个trace object时，trace信息就被触发输出，一直持续到exception被catch住为止，如果异常又被重新抛出，trace信息又会继续输出。因此从最初的异常抛出点到最终异常被真正终结，这段堆栈中的所有trace object都会输出信息，这构成了一个和调用栈对应的trace栈。
如果整个系统纯粹地基于异常传递错误，那么这段代码就足以保证trace栈的完整。而如果在一个混杂模式的系统，即在调用栈中某些调用支持异常，而某些会终结异常转为返回值，那么trace栈就会不完整。 办法当然有:

构造一个per thread的栈（此处用vector实现），每个trace object构造时将自己压入栈，析构时弹出。
trace输出策略调整为，当异常抛出后，在unwinding到第一个trace object时，将trace栈内所有对象依次打印，然后标记状态，后续unwinding的对象不再重复打印，一直到有新的trace object被构造，通常这表明异常已被终结。
这个机制也同样适用于纯返回值模式的系统，将析构函数中打印部分的实现挪到一个公共接口中，在出错时显式调用该接口。

trace输出也可以发生在构造函数中，这就带来的trace object的第二个功能，即实时地输出trace来反映运行轨迹，实际上这就是传统的trace功能。我们可以在构造传入一个控制对象的引用，通过该对象来控制是否在构造时输出。

trace对象的构造成本其实很低，只是初始化一下成员变量，最多再把自己的指针压入栈，而vector栈可以预先resize到足够大，这样就避免了因为动态调整vector大小而引起的动态内存操作。

相比前面提到的语法糖手法，trace object实现了将trace与错误判断分离，ENSURE不再需要后跟一堆上下文输出，这对于代码的简洁和易读性大有裨益。
